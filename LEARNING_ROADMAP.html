✅ 推荐做法

以 NL2SQL BI Agent 为锚点，每学一个知识点，立即在这个项目里改一处代码。每天都能看到项目变好。

学的就是用的，用的就是练的。用输出倒逼输入，用正反馈维持动力。

❌ 避免做法

先刷完所有视频/文档再动手——你会发现前学后忘，没有肌肉记忆。

不追求「完整掌握」再行动：先跑通，再理解，最后优化。学习曲线要短，成就感要快。

📅 学习计划
W1
Week 1 · Python 速通 + LLM API 初体验
目标：跑通项目，理解 LLM 如何「听话」
14 天 · 2h/d
▼
1
环境搭建 + 跑通第一个 LLM 调用
安装 Python 3.10+，用 OpenAI SDK（或 Ollama）发一条 API 调用，理解 Completion vs Chat 接口。尝试换不同的 system prompt 观察回答差异。
安装配置 LLM API
⏱ ~2h
2
Python 核心语法（Java 程序员重点）
重点学：装饰器、上下文管理器（with）、异步（async/await）、字典/列表推导式。类比 Java：@注解 → Python @decorator；Spring Bean → Python 单例；接口 → Protocol。
Python 语法 对比学习
⏱ ~2h
3
Prompt Engineering 核心原则
学习 5 大原则：明确性、分解任务、Few-shot 示例、角色设定、输出格式约束。在项目中修改 SYSTEM_PROMPT，看效果差异。重点：LLM 不是「思考」，是「模式补全」。
Prompt 工程 nl2sql_agent.py
⏱ ~2h
4
理解 Embedding：让 AI 读懂语义
理解 Embedding 本质：把文本转成向量，语义相似 → 向量相近。学习使用 OpenAI/HuggingFace 的 embedding 接口。在项目中找到 embedding.py，理解它如何把 Schema 转成向量存入 Milvus。
Embedding 原理 embedding.py
⏱ ~2h
5
向量数据库：Milvus / ChromaDB
理解为什么需要向量数据库（海量向量快速相似搜索）。Milvus Lite 是零配置版本，正好适合学习。跑通项目的 RAG 检索流程：用户问题 → Embedding → Milvus 查询 → 返回上下文。
向量数据库 retriever.py
⏱ ~2h
6
RAG 核心原理 + 项目检索链路分析
完整走一遍项目 RAG 流程：问题 → Embed → 向量检索 → 上下文组装 → LLM 生成。找到 retriever.py 中的三路检索（Schema + 指标 + SQL 示例），理解各自的作用。
RAG 原理 retriever.py
⏱ ~2h
7
Week1 总结 + 在项目中加一个小功能
建议加一个功能：在 RAG 检索时打印命中的 Schema 片段（方便调试）。这周学到的核心：LLM 是「模式补全」，Embedding 是「语义匹配」，RAG 是「给 LLM 喂正确上下文」。
实践 周总结
⏱ ~2h · 里程碑
🏆 Week1 项目里程碑
跑通完整查询流程
从前端提问到看到图表，全链路跑通，理解数据流向
改一处 Prompt 看效果
修改 SYSTEM_PROMPT，让生成的 SQL 更准确，观察差异
W2
Week 2 · LangGraph 状态机 + Agent 核心概念
目标：理解 Agent 如何「编排」思考步骤
14 天 · 2h/d
▼
1
理解 Agent 本质：循环 + 工具 + 记忆
Agent = LLM + 工具（Tool）+ 循环（Loop）+ 记忆（Memory）。用 Java 类比：Agent 像一个 Spring Bean，根据用户请求（State）决定下一步做什么（路由），每个步骤是一个 Service。
Agent 原理
⏱ ~2h
2
LangGraph 状态机：概念入门
理解 StateGraph、Node、Edge、ConditionalEdge。用生活类比：状态机就像餐厅流程：接单 → 做菜 → 上菜 → 结账，每步有固定动作，失败时回到上一步。重点：它解决了「Agent 步骤编排」的问题。
LangGraphnl2sql_agent.py
⏱ ~2h
3
深度阅读 nl2sql_agent.py 状态图
对照代码画出完整状态图：classify_intent → retrieve → generate → validate → execute → chart。理解每个节点的输入/输出，以及 state 是如何在节点间传递的。重点：State 类型定义。
代码阅读nl2sql_agent.py
⏱ ~2h
4
Self-Correction 循环：让 Agent 自我纠错
今天学的核心：EXPLAIN 校验 + 重试机制，这是 Agent 自我纠错的经典模式。在项目中找到 _route_after_validation 和 _route_after_execution，理解失败时如何回到生成节点。
Self-Correction重试逻辑
⏱ ~2h
5
SQL Executor：数据库连接 + SQL 注入防护
理解 SQLAlchemy 的 text() 防注入机制（参数化查询）。对比 Java JDBC 的 PreparedStatement，两者原理一致。在项目中找到 sql_executor.py，理解 execute 和 validate_with_explain 的区别。
数据库sql_executor.py
⏱ ~2h
6
流式输出 SSE：FastAPI + Server-Sent Events
理解 SSE 和 WebSocket 的区别（SSE 单向，服务→客户端推送）。在项目中找到 routes.py 的 /query/stream 接口，对比 Java 的 Flux / SSE 实现。
SSEroutes.py
⏱ ~2h
7
Week2 总结 + 给 Agent 加一个新节点
建议：在状态图中加一个「结果解释」节点，用 LLM 生成自然语言总结分析本次查询结果（类似 ChatBI 的解释能力）。这周核心：Agent = 状态机 + 工具调用 + 自我纠错。
实践周总结
⏱ ~2h · 里程碑
🏆 Week2 项目里程碑
画出完整状态图
手绘或 Mermaid 画出 Agent 的 6 个节点 + 路由逻辑，能向别人讲清楚
添加一个 Agent 节点
例如：在 generate 前加「SQL 优化建议」节点，或在 chart 后加「自然语言解读」节点
W3
Week 3 · RAG 进阶 + 前沿技术
目标：掌握生产级 RAG 优化方法
14 天 · 2h/d
▼
1
Chunk 策略：如何分块比什么更重要
理解固定窗口、语义段落、层级分块三种策略。分析项目中的 Schema 分块方式（表级 vs 字段级），理解为什么两种都要。在 schema_indexer.py 中添加一种新的分块策略。
Chunkschema_indexer.py
⏱ ~2h
2
混合检索：向量 + BM25 关键词
纯向量检索对专有名词（如列名、表名）不友好，BM25 关键词检索可以弥补。理解 RRF（Reciprocal Rank Fusion）融合方法。项目当前是纯向量检索，可以尝试加上 BM25。
混合检索retriever.py
⏱ ~2h
3
重排序（ReRanking）：精排优于粗排
理解两阶段检索：粗排（向量 ANN）→ 精排（Cross-Encoder ReRanker）。Cohere Rerank 是最流行的方案。尝试给项目加一个 ReRanker，将 Top20 检索结果重排为 Top5。
ReRanking可选升级
⏱ ~2h
4
Function Calling / Tool Use：让 Agent 调用工具
这是 2024 年最实用的 Agent 能力。理解 OpenAI Function Calling 格式，对比 LangGraph 的 Tool 机制。在项目中，Agent 其实就是在做 Function Calling（调用 SQL Executor）。
Function Calling
⏱ ~2h
5
多租户安全：RAG 中的数据隔离
理解 RAG 场景下的多租户问题：不同租户的 Schema 相同，但数据不同。Agent 用 org_id 过滤，这是最简单的方案。进阶：租户专属 Embedding 模型（太贵，不推荐）。
安全架构routes.py
⏱ ~2h
6
评估体系：如何衡量 RAG 质量
学习 RAG 三指标：Context Precision / Recall + Answer Correctness。学习用 LLM-as-Judge 做自动化评估（让 GPT-4 评估另一个 GPT-4 的回答）。为项目写一个简单的 SQL 正确性评估脚本。
RAG 评估脚本开发
⏱ ~2h
7
Week3 总结 + RAG 优化实战
本周核心：RAG 质量 = Chunk 策略 × 检索质量 × 上下文利用率。动手优化一个环节（推荐：混合检索 + BM25）。这周的内容已经是 2025-2026 年生产级 RAG 的标配。
实践周总结
⏱ ~2h · 里程碑
🏆 Week3 项目里程碑
混合检索上线
给 RAG 加上 BM25 关键词召回，用 RRF 融合向量和 BM25 结果
backend/rag/retriever.py
SQL 正确性评估脚本
用 LLM-as-Judge 自动评估生成的 SQL 是否正确统计了正确指标

Week 4 · 前沿扩展 + 项目收尾
目标：接触最新技术方向，确定个人技术栈
14 天 · 2h/d
▼
1
MCP（Model Context Protocol）：Agent 互联标准
MCP 是 2024 年底最火热的协议，让 Agent 可以调用任何外部工具（数据库、API、文件系统）。理解它的 Host/Client/Server 架构。这和 Java 的 SPI 机制有异曲同工之处。
MCP 前沿
⏱ ~2h
2
Memory / 长期记忆：让 Agent 记住上下文
理解 Agent Memory 三层：Working Memory（State）、Session Memory（会话）、Long-term Memory（向量数据库持久化）。用向量数据库存历史对话，实现「记住上次查询」的能力。
Agent Memory
⏱ ~2h
3
LLM Observability：LangSmith / OpenTelemetry
理解 LLM 可观测性三件套：Trace（链路追踪）+ Metrics（指标）+ Logging（日志）。用 LangSmith 追踪 Agent 的每个节点耗时和 token 消耗。对比 Java 的 SkyWalking / Jaeger。
可观测性
⏱ ~2h
4
本地模型：Ollama / vLLM / LM Studio
理解推理框架的区别：Ollama（易用）vs vLLM（高性能）vs LM Studio（GUI）。尝试用 Ollama 跑一个 qwen2.5:14b 或更大的模型，对比和 GPT-4o 的效果差距。
本地推理安装配置
⏱ ~2h
5
项目总结：性能优化 + 工程化
优化方向：LLM 响应缓存（避免重复生成相同 SQL）、连接池优化、向量检索 TopK 调参。理解「AI Native 应用」和传统应用的区别：传统是确定逻辑，AI 是概率输出。
性能优化

🚀 2025-2026 前沿技术方向（选学）
Agentic RAG
Agent 自主决定检索策略、次数和深度，不是单次 RAG，而是多轮「检索→判断→再检索」
MCP（Model Context Protocol）
Anthropic 主导的 Agent 工具调用标准协议，让不同 Agent 可以互操作，类似 USB-C 的 AI 版本
Memory / Long-Term Memory
长期记忆：向量数据库存对话历史，Rerank 找回相关记忆，实现跨会话上下文
vLLM / SGLang（推理加速）
生产级高并发推理框架，PagedAttention 显存管理，支撑企业级 Agent 服务
Text-to-SQL 前沿模型
Groa SQL 等专用模型 + 微调方案，通用 LLM 生成 SQL 仍有 30-40% 错误率
LLM-as-Judge 评估
用强模型评估弱模型输出，自动化 RAG/Agent 质量评估，降低人工标注成本
GraphRAG / KG-RAG
用知识图谱增强 RAG，解决跨文档的复杂推理问题（如「A 和 B 有什么关系」）
Cursor / Windsurf / CoPilot
AI 编码工具本身就是 Agent 应用，学习它们的 Prompt 设计是很好的案例
